# CS-230

The client was The Gaming Room, a company that wanted to expand its game, Draw It or Lose It, into a web-based application. The goal was to create a multiplayer game that supports multiple users and teams playing simultaneously across different operating systems and devices. The software needed to manage game creation, team management, player tracking, and prevent duplicate game instances while maintaining consistent data across user sessions. The application also needs to be scalable, secure, and easy to maintain as the number of users grows.
I think I did a good job evaluating the different operating platforms and explaining why Linux is the best choice for hosting the application. I compared Linux, Windows, macOS, and mobile devices by discussing their strengths and weaknesses for both server-side and client-side development. I also connected my recommendations to The Gaming Room's specific requirements instead of simply describing each operating system. My recommendations included storage management, memory management, networking, security, and scalability, which resulted in a complete software design.
Working through the design document helped me think about the application before writing any code. Creating the design first allowed me to identify the client's requirements, determine the system architecture, and choose the most appropriate technologies. It also helped me understand how the different components, such as the GameService class, teams, players, and the singleton pattern, would work together. Having a clear design made it easier to understand how the software should function and reduced the likelihood of problems during development.
If I could revise one section, I would expand the System Architecture View. Since it was not required for the project, I left it mostly as a placeholder. If I were improving the document, I would include an architecture diagram showing the relationship between clients, the web server, REST APIs, the database, and cloud services. Adding this visual representation would make the design easier to understand and better demonstrate how the application's components communicate with one another.
I interpreted the client's needs by focusing on scalability, reliability, security, and cross-platform compatibility. Since The Gaming Room wanted a web-based application that could support many users simultaneously, I recommended a Linux-based cloud server using a client-server architecture with REST APIs. I also included recommendations for secure authentication, database management, efficient memory usage, and reliable network communication. Considering the user's needs is essential because software should solve the client's problems while remaining easy to use, secure, and capable of supporting future growth. Designing around user requirements helps ensure the final product meets expectations and performs reliably.
I approached the software design by first identifying the client's functional and technical requirements, then evaluating different operating platforms before selecting the best solution. I considered factors such as performance, scalability, security, storage management, memory management, and networking before making my recommendations. In the future, I would continue using a structured design process that includes gathering requirements, creating UML diagrams, evaluating multiple design options, documenting architectural decisions, and considering security and scalability early in development. This project reinforced the importance of planning before coding and showed how a strong software design leads to a more organized and maintainable application.
